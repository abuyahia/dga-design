Accessibility Notes

Derived from chip.contract.md §Aria Contract and audit rules CHIP-A11Y-001 through CHIP-A11Y-004.

Rule Requirement Implementation
Accessible label
CHIP-A11Y-001 		Every chip must have a non-empty accessible name. Provide <span class="chip__label">Label</span> as a child, or set aria-label on the root.
Semantic element
CHIP-STRUCT-001 		Interactive chips require a native interactive element. Use <button type="button"> for toggle/filter chips. Use <span> only for display-only (non-interactive) chips.
Selected state
CHIP-A11Y-003 		Toggle/selectable chips must communicate selection to assistive technology. Standalone toggle: aria-pressed="true|false". Group selection: aria-selected="true|false" inside role="group". Never use both on the same chip.
Disabled patterns Two valid disabled mechanisms with different keyboard behaviour. disabled attribute removes chip from tab order. aria-disabled="true" keeps chip in tab order (useful for discoverable disabled states with tooltips).
Focus ring
CHIP-A11Y-002 		Focus must be visually indicated. outline: none on the root is set but replaced by ::after ring. Focus ring is rendered via .chip:focus::after with position: absolute; inset: var(--chip-focus-inset). Must not be overridden without equivalent replacement.
Touch target
CHIP-A11Y-004 		Minimum 24×24px touch target. Large (32px) ✓ and Medium (24px) ✓ pass. Small (20px) fails. Requires design team exception or height increase before shipping on touch interfaces.
RTL
CHIP-RTL-001 		Chip must render correctly in right-to-left contexts. chip.css uses padding-inline (logical property). No physical directional properties. RTL is applied via [dir="rtl"] on a container — no chip-specific class needed.
Group pattern Filter chips in a group require a container with accessible group semantics. Wrap in <div role="group" aria-label="[purpose]">. Keyboard arrow-key navigation within the group is not provided by chip.css — implement via JavaScript if required.

Known Implementation Deviations

Intentional differences between the Figma design source and this implementation. All deviations are documented in analysis.md and chip.contract.md.

DEV-1 · Focus outline dimensions
Dynamic inset vs. Figma fixed pixel widths
Figma models the focus outline as a fixed-size RECTANGLE node: Small 55×28px, Medium 59×32px, Large 68×40px. These values assume the chip label is always "Item" and break when label length changes. Implementation uses inset: var(--chip-focus-inset) (resolves to −4px) which grows with the chip's natural width and height, supporting dynamic labels. Visual result is identical to Figma for the "Item" label.
/* chip.css */ .chip--primary:focus::after { content: ""; position: absolute; inset: var(--chip-focus-inset); /* = -4px; dynamic, not fixed */ border-radius: var(--chip-radius-default); border: var(--chip-focus-outline-width) solid var(--chip-focus-outline-color); pointer-events: none; }
DEV-2 · Cross-component token reference in Figma
Primary/Pressed text color borrows a Button-namespaced token
The Figma source uses --Button-button-background-primary-pressed as the text color for the Primary/Pressed chip state. This is a Button-scoped background token being repurposed as text in a different component — a cross-component token leak. chip.css proposes the correct chip-scoped replacement: --chip-text-primary-pressed: var(--sa-900). Awaiting confirmation from design team (see analysis.md §KI-6).
DEV-3 · Figma source typo in token name
On-color disabled background token has a spelling error in Figma
The Figma design file contains the token name --Chip-chip-background-on-color-diabled (missing 's' in "disabled"). All generated files use the corrected name --chip-on-color-bg-disabled. The Figma source token should be renamed to match. See analysis.md §KI-7.
DEV-4 · Reversed font-weight progression
Small chips use heavier font-weight (600) than Medium/Large (500)
Typically, larger sizes carry heavier weights. The Figma source inverts this: Small = font-weight: 600 (Semibold), Medium = 500, Large = 500. Preserved as-is from Figma. Design team should confirm this is intentional. See analysis.md §Architecture Findings.
DEV-5 · Size-invariant padding
All three sizes use the same padding-inline (12px)
Typically padding scales with size. In this component, all sizes share padding-inline: var(--chip-padding-inline) (maps to 12px / --spacing-md). Only height changes across sizes. Preserved as-is from Figma. Design team should confirm at Large size. See analysis.md §KI-9 for the related Figma token naming discrepancy.
